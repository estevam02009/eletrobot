# 🏛 Arquitetura do Sistema

## Visão Geral

O **EletroBot AI** é uma plataforma de atendimento inteligente desenvolvida para lojas de eletrodomésticos.

Seu objetivo é automatizar o atendimento ao cliente utilizando Inteligência Artificial, integração com WhatsApp e um catálogo inteligente de produtos.

A arquitetura foi planejada para ser escalável, organizada e preparada para futuras integrações.

---

# Objetivos da Arquitetura

* Separação de responsabilidades.
* Facilidade de manutenção.
* Escalabilidade.
* Baixo acoplamento.
* Alta coesão.
* Código reutilizável.
* Facilidade para testes.

---

# Arquitetura em Camadas

```text
Cliente
    │
    ▼
API REST
    │
    ▼
Controllers
    │
    ▼
Services
    │
    ▼
Repositories
    │
    ▼
Models
    │
    ▼
MySQL
```

Cada camada possui uma responsabilidade específica.

---

# Controllers

Responsáveis por:

* Receber requisições HTTP.
* Validar Requests.
* Chamar os Services.
* Retornar Resources.

Os Controllers **não devem conter regras de negócio**.

---

# Services

Os Services concentram todas as regras de negócio.

Exemplos:

* Cadastro de Produtos
* Atualização de Estoque
* Criação de Pedidos
* Atendimento do ChatBot

---

# Repositories

Os Repositories são responsáveis pelo acesso aos dados.

Exemplos:

* Consultar produtos
* Buscar clientes
* Filtrar pedidos

---

# Models

Representam as entidades do sistema.

Exemplos:

* Product
* Category
* Brand
* Customer
* Order

---

# API Resources

Todos os retornos da API utilizarão Resources.

Benefícios:

* Padronização.
* Segurança.
* Facilidade de manutenção.

---

# Requests

Toda validação será realizada utilizando Form Requests.

Exemplos:

* StoreProductRequest
* UpdateProductRequest

---

---

# Banco de Dados

O banco de dados utilizará MySQL.

Os principais módulos serão:

* Produtos
* Categorias
* Marcas
* Estoque
* Clientes
* Conversas
* Pedidos

---

# Integrações

A plataforma será preparada para integração com:

* WhatsApp
* OpenAI
* APIs de pagamento
* Sistemas ERP

---

# Padrões Utilizados

* MVC
* Service Layer
* Repository Pattern (quando necessário)
* Form Requests
* API Resources
* RESTful API
* Versionamento Semântico
* Conventional Commits

---

# Escalabilidade

A arquitetura foi planejada para permitir a evolução do sistema para um modelo SaaS (Software as a Service), suportando múltiplas empresas e integrações adicionais sem necessidade de grandes alterações estruturais.
